# 多人合盤交互比較命盤 — 設計計劃

## 現狀分析

目前 PurpleStar 是一個純前端的紫微斗數排盤應用：
- 使用 iztro 計算命盤，Zustand + localStorage 儲存
- 支援多張命盤儲存，但只能**逐張檢視**，無法並排比較
- AI 對話僅針對**單一命盤**進行解讀
- 無任何合盤、配對、多人互動功能

---

## 功能目標

讓使用者能夠：
1. **選取 2～N 張命盤**進行合盤比較
2. **並排檢視**多人宮位、星曜、四化差異
3. **透過 AI 對話**詢問多人之間的關係解析（感情、事業合作、親子等）
4. **儲存合盤組合**方便日後回顧

## 設計方案

### Phase 1：資料模型擴展

#### 新增 types

```typescript
// 家庭角色
export type FamilyRole =
  | 'self'       // 本人
  | 'spouse'     // 配偶
  | 'father'     // 父親
  | 'mother'     // 母親
  | 'son'        // 兒子
  | 'daughter'   // 女兒
  | 'sibling'    // 兄弟姐妹
  | 'partner'    // 合作夥伴
  | 'friend'     // 朋友
  | 'other'      // 其他

// 合盤成員（帶角色標注）
export interface ComparisonMember {
  chartId: string
  role: FamilyRole
  label?: string                  // 自訂標籤，如「大女兒」「小女兒」
}

// 合盤組合
export interface ChartComparison {
  id: string
  name: string                    // 使用者自訂名稱，如「我們一家四口」
  members: ComparisonMember[]     // 帶角色的成員列表（取代原本的 chartIds）
  createdAt: Date
  updatedAt: Date
}
```

> **設計重點：不再用單一 `ComparisonType`**
>
> 原本的設計用一個 type 定義整組的關係（感情/事業/家庭），但一個家庭群組內有多種關係。
> 改為**每個成員標注角色**，AI 可以自動推導出所有關係對：
> - 爸爸 × 媽媽 → 夫妻關係
> - 爸爸 × 大女兒 → 父女關係
> - 媽媽 × 小女兒 → 母女關係
> - 大女兒 × 小女兒 → 姐妹關係
> - ...共 6 組

#### Store 擴展

在 Zustand store 中新增：

```typescript
// 合盤管理
comparisons: ChartComparison[]
currentComparison: ChartComparison | null
addComparison: (comparison: ChartComparison) => void
updateComparison: (id: string, updates: Partial<ChartComparison>) => void
deleteComparison: (id: string) => void
setCurrentComparison: (comparison: ChartComparison | null) => void
```

persist 的 `partialize` 加入 `comparisons`。

### Phase 2：合盤比較頁面 UI

#### 路由

新增路由 `/#/compare`，對應 `ComparisonPage.tsx`。

#### 頁面流程

```
首頁/導航列 → [合盤比較] 入口
    ↓
合盤列表頁（已儲存的合盤 + 新增按鈕）
    ↓
建立合盤：
  1. 輸入合盤名稱
  2. 選擇類型（感情/事業/家庭/自訂）
  3. 從已存命盤中勾選 2+ 張（或即時新排盤）
  4. 確認建立
    ↓
合盤檢視頁
```

#### 合盤檢視頁 — 核心 UI 設計

採用**標籤頁 + 並排**的混合佈局：

**Tab 1：命盤總覽**
- 上方顯示所有參與者的基本資訊卡片（姓名、生日、五行局、命宮主星）
- 下方可切換檢視各人的完整 12 宮格命盤（點擊切換，而非同時顯示多個宮格，避免畫面過擠）

**Tab 2：宮位對照表**
- 一張表格，列 = 12 宮位名稱，行 = 各參與者
- 每格顯示該宮的主星 + 亮度 + 四化
- 相同主星用顏色高亮，方便一眼看出共同點
- 支援點擊展開單一宮位的詳細比較

**Tab 3：星曜對照**
- 以 14 主星為列，各參與者為行
- 顯示每顆星落在哪個宮位 + 亮度 + 四化
- 快速比較同一顆星在不同人盤中的落宮差異

**Tab 4：四化對照**
- 比較各人的生年四化、大限四化
- 交叉分析：A 的化忌落在 B 的哪個宮位（關係影響）

**Tab 5：AI 合盤對話**（見 Phase 3）

### Phase 3：AI 多人合盤對話

#### Prompt 設計

在現有 `formatChartForAI()` 基礎上，新增 `formatComparisonForAI()`：

```typescript
function formatComparisonForAI(
  charts: Chart[],
  comparisonType: ComparisonType
): string {
  // 1. 列出每個人的完整命盤摘要（複用現有 formatChartForAI）
  // 2. 加入合盤類型說明
  // 3. 加入交叉分析重點提示：
  //    - 雙方命宮、夫妻宮、官祿宮的星曜互動
  //    - 四化飛星的交叉影響
  //    - 大限流年的時機配合
  return prompt
}
```

#### System Prompt 擴展

在 `buildSystemPrompt()` 中，當偵測到合盤模式時：

```
你正在分析一組合盤（${comparisonType}關係）。
請綜合以下 ${n} 人的命盤資料，分析他們之間的互動關係。

重點分析方向：
- 各人命宮主星的性格互補或衝突
- 夫妻宮/官祿宮/財帛宮的星曜配對
- 四化飛星的交叉影響（A 的化忌落在 B 的哪個宮位）
- 大限流年的時機契合度
- 根據關係類型（感情/事業/家庭）給出針對性建議
```

#### 預設問題模板

根據 `ComparisonType` 提供不同的預設問題：

**感情配對 (romance)：**
- 我們的感情配對度如何？
- 相處上會有哪些摩擦？
- 最適合我們的相處模式是什麼？

**事業合作 (business)：**
- 我們適合一起合作嗎？
- 各自適合擔任什麼角色？
- 合作的最佳時機是什麼時候？

**親子家庭 (family)：**
- 親子之間的互動模式是什麼？
- 教養上需要注意什麼？
- 如何改善彼此的溝通？

### 關鍵互動細節

| 情境 | 處理方式 |
|------|---------|
| 使用者只有 1 張命盤就想合盤 | 建立合盤時提示「至少需要 2 張命盤」，並提供「即時排新盤」按鈕 |
| 使用者從排盤頁直接想合盤 | 已存命盤列表右側加「合盤」icon，點擊後帶該盤跳轉到 `/compare?preselect=chartId` |
| 合盤中某張盤被刪除 | 合盤檢視時檢查 members 的 chartId 是否都存在，缺少的顯示「此命盤已刪除」提示 |
| AI 未設定 API Key | 合盤 Tab 5 顯示與現有相同的「請先設定 AI」引導頁面 |
| 手機版宮位對照表太寬 | 提供「篩選關係對」按鈕只看兩人；或改為垂直卡片式佈局 |
| 3+ 人合盤 | 表格自動增加欄，AI prompt 依序列出所有人的命盤 |
| 同角色多人（如兩個女兒） | 用 `label` 欄位區分（大女兒/小女兒），AI prompt 中帶入標籤 |
| 使用者問特定兩人的關係 | AI 根據角色自動推導關係類型，聚焦對應宮位（子女宮 vs 父母宮等） |
| 家庭合盤問整體運勢 | AI 綜合四人流年，找出共同吉凶時段 |
| 小孩年紀很小（如 7 歲） | AI 解讀時考量年齡，側重性格培養而非事業財運 |

## 技術注意事項

1. **iztro `_astrolabe` 的利用**：目前 Chart 中保留了 `_astrolabe` 參考，可利用 iztro 的 API 做更深入的交叉分析（如飛星到指定宮位）。但 `_astrolabe` 不被 persist，需要在載入合盤時重新計算。

2. **效能考量**：合盤涉及多張命盤同時在記憶體中，但紫微命盤的資料量不大（每張約幾 KB），即使 5-6 人合盤也不會有效能問題。

3. **localStorage 容量**：多張命盤 + 多組合盤 + 對話記錄，需注意 localStorage 的 5MB 上限。可考慮：
   - 對話記錄超過一定長度時自動截斷舊訊息
   - 或未來遷移至 IndexedDB

4. **RWD 適配**：宮位對照表在手機上需要水平滾動或改為卡片式佈局，避免表格過寬。

5. **合盤人數上限**：建議 UI 上限制最多 4-5 人，超過後表格和 AI prompt 都會變得過長難讀。

6. **AI Token 消耗**：多人合盤的 prompt 會比單人長數倍，需在 prompt 中做適度精簡，或提示使用者這會消耗更多 API 額度。
